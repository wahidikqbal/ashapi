## 🔴 KRITIS (5 Issues)

### K1. Dual Response Bug di `success/4` — JSON + Redirect Bersamaan

**File:** `lib/ashapi_web/controllers/auth_controller.ex:25-34`

**Deskripsi:** Fungsi `success/4` memanggil `json()` (mengirim response JSON) lalu dilanjutkan dengan `redirect()` (mengirim response redirect). Di Plug.Conn, hanya response pertama yang terkirim — sisanya akan raise `Plug.Conn.AlreadySentError`.

**Kode bermasalah:**
```elixir
conn
|> delete_session(:return_to)
|> store_in_session(user)
|> put_resp_cookie("token", token, ...)
|> json(%{success: true, message: message, token: token})   # ← sends response
|> assign(:current_user, user)                               # ← never applied
|> put_flash(:info, message)                                 # ← never applied
|> redirect(to: return_to)                                   # ← raises error
```

**Dampak:** Callback `success/4` gagal total. Browser auth flow tidak berfungsi. User tidak di-redirect, flash message hilang.

**Rekomendasi:** Pisahkan logika untuk browser vs API. `success/4` hanya untuk browser (redirect saja). API login via `login/2`.

---

### K2. `failure/3` Juga Dual Response — Redirect + JSON

**File:** `lib/ashapi_web/controllers/auth_controller.ex:54-61`

**Deskripsi:** Sama seperti K1, `failure/3` memanggil `redirect()` lalu `json()`.

**Kode bermasalah:**
```elixir
conn
|> put_flash(:error, message)
|> redirect(to: ~p"/sign-in")           # ← sends 302
|> put_status(:unauthorized)            # ← irrelevant
|> json(%{error: "Incorrect email..."}) # ← raises error
```

**Dampak:** Tidak ada response JSON yang terkirim, flash message juga hilang.

**Rekomendasi:** Sama seperti K1 — satu fungsi, satu response.

---

### K3. Token Lifetime Hanya 1 Menit

**File:** `lib/ashapi/accounts/user.ex:12`

**Deskripsi:** Token JWT expired dalam 1 menit. Bahkan ada komentar TODO untuk mengubahnya.

```elixir
token_lifetime {1, :minutes} # ubah menjadi {24, :hours} untuk 24 jam
```

**Dampak:** User akan kehilangan autentikasi setiap 1 menit. Harus login lagi terus-menerus. Sesi browser akan expire sangat cepat.

**Rekomendasi:** Ubah ke `{24, :hours}` untuk production. Untuk development bisa `{60, :minutes}` atau `{24, :hours}`.

---

### K4. Inkon sistensi Cookie `max_age`

**File:** `lib/ashapi_web/controllers/auth_controller.ex:23` dan `:94`

**Deskripsi:** Dua path login memberikan umur token cookie berbeda:

| Path | File | max_age |
|------|------|---------|
| Browser auth callback `success/4` | auth_controller.ex:23 | **60** (1 menit) |
| Custom API `login/2` | auth_controller.ex:94 | **86.400** (24 jam) |

**Dampak:** User yang login via browser callback hanya punya token 1 menit, sedangkan via API punya 24 jam. Membingungkan dan inkonsisten.

**Rekomendasi:** Samakan nilai `max_age` dengan `token_lifetime` yang benar.

---

### K5. `secure: false` Hardcoded untuk Cookie

**File:** `lib/ashapi_web/controllers/auth_controller.ex:21` dan `:88`

**Deskripsi:** Flag `secure` untuk cookie diset `false` secara hardcoded, dengan komentar "pada saat deploy, bisa diatur ke false saat development".

```elixir
put_resp_cookie("token", token,
  path: "/",
  http_only: true,
  secure: false,       # ← harusnya pakai config environment
  same_site: "Lax",
  max_age: 60
)
```

**Dampak:** Di production, cookie token dikirim via HTTP (tidak hanya HTTPS). Rentan terhadap man-in-the-middle attack.

**Rekomendasi:** Gunakan `Application.compile_env(:ashapi, :env) == :prod` untuk set `secure` secara otomatis.

## 🟠 HIGH (13 Issues)

### H1. `.env` Tidak Ada di `.gitignore`

**File:** `.gitignore`

**Deskripsi:** File `.env` berisi secrets (database URL, signing secrets) tapi tidak tercantum di `.gitignore`. Sangat berisiko tercommit ke repository publik.

**Rekomendasi:** Tambahkan `.env` ke `.gitignore`.

---

### H2. Secrets Development & Test Hardcoded di Version Control

**File:** 
- `config/dev.exs:27` — `secret_key_base` hardcoded
- `config/dev.exs:72` — `token_signing_secret` hardcoded
- `config/test.exs:2` — `token_signing_secret` hardcoded

**Deskripsi:** Tiga secrets penting disimpan sebagai string literal dalam file konfigurasi yang ter-version control.

**Dampak:** Jika repository publik, secrets bisa diekspos. Siapa pun yang tahu dev secrets bisa memalsukan JWT token.

**Rekomendasi:** Gunakan environment variables atau file `.env` yang di-load di `runtime.exs`. Alternatif: gunakan placeholder dan dokumentasikan cara generate.

---

### H3. API Login Tidak Set Session / Current User

**File:** `lib/ashapi_web/controllers/auth_controller.ex:73-109`

**Deskripsi:** Fungsi `login/2` hanya set cookie token, tidak memanggil `store_in_session(user)` dan tidak assign `current_user` ke conn.

```elixir
def login(conn, params) do
  # ... sign in logic ...
  conn
  |> put_resp_cookie("token", token, ...)  # OK
  |> json(%{success: true, ...})            # OK
  # Missing: store_in_session(user)
  # Missing: assign(conn, :current_user, user)
end
```

**Dampak:** `GET /api/auth/me` hanya bisa membaca user dari cookie (via AuthPlug). Jika AuthPlug gagal, session kosong.

**Rekomendasi:** Tambahkan `store_in_session(user)` dan `assign(conn, :current_user, user)` setelah login.

---

### H4. Email Sender Masih Pakai `noreply@example.com`

**File:**
- `lib/ashapi/accounts/user/senders/send_password_reset_email.ex:17`
- `lib/ashapi/accounts/user/senders/send_new_user_confirmation_email.ex:17`

**Deskripsi:** Kedua email sender menggunakan alamat `noreply@example.com` dengan comment TODO.

```elixir
# TODO: Replace with your email
|> from({"noreply", "noreply@example.com"})
```

**Dampak:** Email yang terkirim akan gagal SPF/DKIM validation. Bisa masuk spam atau tidak terkirim sama sekali.

**Rekomendasi:** Jadikan from address sebagai konfigurasi aplikasi (Application config atau env var).

---

### H5. Password Reset URL Mungkin Tidak Cocok dengan Router

**File:** 
- `lib/ashapi/accounts/user/senders/send_password_reset_email.ex:25`
- `lib/ashapi_web/router.ex:82-86`

**Deskripsi:** Email sender mengirim link `~p"/password-reset/#{token}"`. Router menggunakan `reset_route` macro dari AshAuthentication.Phoenix. Perlu diverifikasi apakah URL cocok dengan route yang digenerate.

**Rekomendasi:** Verifikasi path yang digenerate oleh `reset_route` macro. Jika tidak cocok, sesuaikan URL di sender.

---

### H6. Confirm Email URL Mungkin Tidak Cocok dengan Router

**File:**
- `lib/ashapi/accounts/user/senders/send_new_user_confirmation_email.ex:25`
- `lib/ashapi_web/router.ex:89-91`

**Deskripsi:** Sama seperti H5, email confirmation mengirim link `~p"/confirm_new_user/#{token}"`. Router menggunakan `confirm_route` macro.

**Rekomendasi:** Verifikasi path dan sesuaikan.

---

### H7. Order Plug `fetch_cookies` Tidak Optimal

**File:** `lib/ashapi_web/router.ex:18-27`

**Deskripsi:** Pipeline API memanggil `fetch_cookies` dua kali:

```
TokenFromCookie → load_from_bearer → set_actor → fetch_cookies → AuthPlug
```

`TokenFromCookie` secara internal memanggil `fetch_cookies`, lalu di pipeline ada `fetch_cookies` eksplisit.

**Rekomendasi:** Pindahkan `fetch_cookies` sebelum `TokenFromCookie` atau hapus yang duplikat.

---

### H8. AuthPlug Tidak Handle Authorization Header

**File:** `lib/ashapi_web/middlewares/auth_plug.ex:11`

**Deskripsi:** AuthPlug hanya membaca token dari cookie, tidak dari Authorization header.

```elixir
def call(conn, _opts) do
  token = conn.cookies["token"]  # ← hanya cookie
```

**Dampak:** Client yang mengirim Bearer token via header tidak akan terdeteksi oleh AuthPlug. (Tapi ini sebagian teratasi oleh `token_form_cookie.ex` yang inject cookie ke header.)

**Rekomendasi:** Tambahkan fallback ke Authorization header di AuthPlug.

---

### H9. Tidak Ada Konfigurasi Mailer untuk Production

**File:** `config/runtime.exs:130-145`

**Deskripsi:** Runtime config hanya berisi comment contoh Mailgun. Tidak ada implementasi aktif untuk mailer production. Swoosh masih pakai Local adapter di semua environment.

**Dampak:** Email confirmation dan password reset tidak akan terkirim di production.

**Rekomendasi:** Implementasi mailer adapter (Mailgun, SMTP, dll) dengan konfigurasi dari env vars.

---

### H10. Tidak Ada Rate Limiting untuk Auth Endpoints

**File:** `lib/ashapi_web/router.ex:49`

**Deskripsi:** Endpoint `POST /api/auth/login` dan `POST /api/json/users/sign-in` tidak memiliki rate limiting atau proteksi brute force.

**Rekomendasi:** Implementasi rate limiting (misalnya dengan `hammer` atau `ex_rated`) untuk auth endpoints.

---

### H11. CORS untuk Swagger UI

**File:** `lib/ashapi_web/router.ex:57-59`

**Deskripsi:** Swagger UI di `/api/json/swaggerui` mungkin perlu CORS headers jika diakses dari domain berbeda.

**Rekomendasi:** Pastikan Swagger UI path juga mendapat CORS headers.

---

### H12. Token Lifetime vs Cookie max_age Tidak Sinkron

**Deskripsi:** `token_lifetime` di `user.ex` = 1 menit, tapi cookie `max_age` di `auth_controller.ex` = 60 atau 86400. Dua nilai ini harusnya sinkron. Token expired 1 menit tapi cookie diset 24 jam = cookie membawa token expired.

**Rekomendasi:** Selaraskan `max_age` cookie dengan `token_lifetime` resource.

---

### H13. `SameSite` Cookie Antara Session dan Token Berbeda

**File:** 
- `lib/ashapi_web/endpoint.ex:11` — session cookie: `same_site: "Strict"`
- `lib/ashapi_web/controllers/auth_controller.ex:22,93` — token cookie: `same_site: "Lax"`

**Deskripsi:** Session cookie menggunakan `Strict`, token cookie menggunakan `Lax`. Inkonsisten.

**Rekomendasi:** Standardisasi ke `Strict` untuk production, `Lax` untuk development.

## 🟡 MEDIUM (10 Issues)

### M1. Error JSON Tidak Sesuai JSON:API Spec

**File:** `lib/ashapi_web/controllers/error_json.ex:19`

**Deskripsi:** Error response mengembalikan object, tapi JSON:API spec mengharuskan array.

```elixir
# Current (salah):
%{errors: %{detail: "Not Found"}}

# JSON:API spec (benar):
%{errors: [%{detail: "Not Found"}]}
```

**Rekomendasi:** Ubah ke format array sesuai JSON:API spec.

---

### M2. Tidak Ada CSRF Protection untuk API Pipeline

**File:** `lib/ashapi_web/router.ex:18-27`

**Deskripsi:** API pipeline tidak memiliki `protect_from_forgery` (hanya browser pipeline yang punya). Ini umum, tapi perlu kesadaran bahwa endpoint API rentan terhadap CSRF jika tidak menggunakan token-based auth dengan benar.

**Rekomendasi:** Dokumentasikan bahwa API hanya mengandalkan token-based auth (JWT Bearer/Cookie) untuk proteksi.

---

### M3. Test Coverage Sangat Minim

**File:** `test/`

**Deskripsi:** Hanya 3 file test untuk controller. Tidak ada test untuk:

- Auth flow (login, register, logout, token)
- Resource actions (CRUD posts, users)
- Policy authorization
- API endpoints (JSON:API)
- CORS configuration
- Error handling

**Rekomendasi:** Tambah test coverage minimal untuk auth flow dan API endpoints.

---

### M4. Seeds Kosong

**File:** `priv/repo/seeds.exs`

**Deskripsi:** File seeds hanya berisi comment template, tidak ada data seed.

**Rekomendasi:** Tambahkan seed data (misalnya user admin, contoh posts).

---

### M5. `accounts.ex` Menggunakan `AshAdmin.Domain` Extension

**File:** `lib/ashapi/accounts.ex:4`

**Deskripsi:** Accounts domain mengaktifkan `AshAdmin.Domain` extension. Ini seharusnya hanya untuk development (AshAdmin cuma aktif di dev), tapi extension tetap di-load.

**Rekomendasi:** Hapus `AshAdmin.Domain` dari domain extension di production, atau pastikan AshAdmin aman digunakan.

---

### M6. `AshapiWeb.AshJsonApiRouter` — Prefix `/api/json` di Dua Tempat

**File:** 
- `lib/ashapi_web/ash_json_api_router.ex:6` — `prefix: "/api/json"`
- `lib/ashapi_web/router.ex:54` — `scope "/api/json"`

**Deskripsi:** Prefix didefinisikan dua kali — di router scope dan di AshJsonApiRouter. Bisa menyebabkan duplikasi path (`/api/json/api/json/...`).

**Verifikasi:** Perlu dicek apakah `forward "/"` di scope `/api/json` dengan router yang punya prefix `/api/json` menghasilkan path yang benar.

**Rekomendasi:** Hapus prefix dari salah satu (biasanya cukup di scope router saja, atau set prefix kosong di json api router).

---

### M7. Password Reset Belum Terverifikasi Fungsional

**Deskripsi:** Fitur password reset menggunakan AshAuthentication bawaan, tapi perlu verifikasi end-to-end bahwa:
1. Token reset terkirim via email
2. Link di email valid
3. Reset password action work

**Rekomendasi:** Test flow reset password secara manual dan tambahkan test.

---

### M8. Tidak Ada Pagination Configuration untuk JSON:API

**Deskripsi:** `default_page_type: :keyset` sudah di config, tapi tidak ada konfigurasi pagination untuk JSON:API response (page size, dll).

**Rekomendasi:** Set default page size untuk JSON:API (misal 25 items/page).

---

### M9. Develop Secret Key Base Terlalu Sederhana

**File:** `config/dev.exs:27`

**Deskripsi:** Secret key base di dev.exs adalah string statis. Untuk development tidak masalah, tapi perlu dicatat bahwa ini BUKAN untuk production.

**Rekomendasi:** Sudah OK untuk development, pastikan tidak terpakai di production.

---

### M10. Tidak Ada Logging untuk Failed Authentication

**Deskripsi:** Auth failure hanya redirect ke sign-in page. Tidak ada logging untuk failed attempt yang bisa digunakan untuk monitoring security.

**Rekomendasi:** Tambahkan logging untuk failed authentication attempts (email, IP, timestamp, reason).
